## Context

本项目是一个抖音创作者内容监控工具，属于内容处理流水线的第一阶段。当前阶段仅实现"监控发现 + 飞书记录"，后续阶段（视频下载、音频转写、内容分析）将在未来迭代中添加。

**约束条件：**
- 监控创作者数量：当前 2 个，未来不超过 20 个
- 监控频率：每 59 分钟一次（由 OpenClaw Cron 调度）
- 运行环境：本地 Windows 机器
- 状态存储：仅使用飞书多维表格，不引入本地数据库
- 技术偏好：成熟、轻量级

**外部服务：**
- TikHub API (`/api/v1/douyin/app/v3/fetch_user_post_videos`) — 获取创作者视频列表
- 飞书开放平台 API — 多维表格读写

## Goals / Non-Goals

**Goals:**
- 独立可运行的监控模块，供 OpenClaw Cron 定时调用
- 通过 TikHub API 拉取指定创作者最新视频列表
- 通过飞书多维表格查询已有记录实现去重
- 新发现的视频自动写入飞书多维表格，填入阶段①相关字段
- 模块可被 import，供将来的 `main.py` 编排使用
- 配置与敏感信息分离管理

**Non-Goals:**
- 不实现视频下载、音频转写、内容分析功能（阶段②③④⑤）
- 不实现内置调度器（由 OpenClaw Cron 负责）
- 不实现飞书消息通知
- 不实现 Web UI 或 API 服务
- 不实现自动创建飞书多维表格

## Decisions

### D1: 模块架构 — 分层职责

```
monitor.py (编排层)
  ├── tikhub.py    (数据源层)
  ├── feishu.py    (存储层)
  ├── config.py    (配置层)
  └── models.py    (模型层)
```

**理由**：单一职责，每个模块独立可测试，且 tikhub/feishu 客户端可在后续阶段复用。

### D2: 去重策略 — 飞书表格查询

每次运行时查询飞书表格获取该博主已有视频 ID 集合，与 TikHub 返回的最新视频做差集。

**替代方案**：本地 SQLite 存储。
**选择理由**：用户明确要求只使用飞书存储，减少本地状态管理复杂度。20 个创作者、59 分钟频率下，飞书 API 调用量完全在限额内。

**查询优化**：按博主名称筛选 + 只取视频 ID 字段，减少数据传输量。

### D3: 配置管理 — YAML + .env 分离

- `config.yaml`：非敏感配置（创作者列表、TikHub 端点、飞书表格标识等）
- `.env`：敏感信息（TikHub API Key、飞书 App ID、飞书 App Secret）

**技术选型**：`pydantic-settings` 加载配置，`python-dotenv` 读取 `.env`，类型安全且自动验证。

### D4: HTTP 客户端 — httpx (同步模式)

使用 httpx 同步模式，当前规模不需要异步。

**替代方案**：requests。
**选择理由**：httpx API 与 requests 兼容但更现代，且未来可无缝切换到异步。

### D5: 日志 — loguru

替代标准库 logging，配置更简洁，输出更可读。

### D6: monitor.py 双模式运行

- **CLI 模式**：`python -m hot_pulse.monitor` 直接执行一轮监控
- **模块模式**：`from hot_pulse.monitor import run_monitor` 供 main.py 调用

### D7: 飞书字段映射

飞书表格字段与写入值映射：

| 字段 | 类型 | 写入值 |
|------|------|--------|
| 任务名 | 文本 | `{视频标题}` |
| 优先级 | 单选 | "中" |
| 状态 | 单选 | "新视频" |
| 任务类型 | 单选 | "视频" |
| 任务创建时间 | 日期时间 | 当前时间 |
| 平台 | 单选 | "抖音" |
| 博主 | 文本 | 创作者名称 |
| 视频链接 | 文本 | 视频URL |
| 视频发现时间 | 日期时间 | 当前时间 |
| 视频 ID | 文本 | 视频唯一ID |
| 视频/音频/转写/分析相关字段 | — | 留空（后续阶段填写） |
| last_update_time | 日期时间 | 当前时间 |

### D8: 项目结构

```
hot_pulse/
├── config.yaml              # 非敏感配置
├── .env                     # 敏感信息 (git忽略)
├── .gitignore
├── pyproject.toml
└── src/
    └── hot_pulse/
        ├── __init__.py
        ├── config.py        # 配置加载
        ├── models.py        # 数据模型
        ├── tikhub.py        # TikHub API 客户端
        ├── feishu.py        # 飞书多维表格客户端
        └── monitor.py       # 监控编排 (独立可运行)
```

## Risks / Trade-offs

- **[TikHub API 不可用]** → 添加重试机制（最多 3 次，指数退避），记录错误日志，不阻塞其他创作者的查询
- **[飞书 API 频率限制]** → 20 个创作者 * 每次约 2 次 API 调用（1 查询 + 1 写入）= 约 40 次/小时，远低于限制。如有风险可加请求间隔控制
- **[飞书查询性能]** → 随着记录增多，查询已有视频 ID 可能变慢。缓解：使用飞书筛选条件限定博主 + 只请求视频 ID 字段
- **[TikHub 返回数据结构变化]** → 数据模型加字段校验，变化时 loguru 报错并跳过该条记录
- **[抖音创作者 sec_uid 失效]** → 配置验证阶段检查，运行时报错并跳过该创作者
